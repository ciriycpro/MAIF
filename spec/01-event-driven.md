# 01. Event-driven архитектура

## Требование

Реализация стандарта **MUST** строиться на событийно-ориентированной архитектуре (Event-Driven Architecture, EDA) как сквозном принципе организации межагентного взаимодействия.

## Обоснование

Событийная архитектура является условием реализуемости остальных пяти требований стандарта:

- Сквозной аудит (раздел 05) опирается на трассируемые события.
- Малая связанность (раздел 03) обеспечивается публикацией и подпиской на события, а не прямыми вызовами.
- Human-in-the-loop в реальном времени (раздел 04) реализуется как асинхронное событие подтверждения.
- Типизированные контракты (раздел 02) описывают структуру событий, а не структуру вызовов.

Помимо этого, событийная декомпозиция является математическим условием схождения задач координации автономных решений при больших объёмах данных и распределённом исполнении.

## Минимальные обязательства реализации

Реализация **MUST** обеспечивать:

1. **Асинхронную доставку сообщений** между агентами. Прямые синхронные вызовы между автономными агентами не допускаются как основной механизм взаимодействия.
2. **Идентификацию событий** уникальными идентификаторами, обеспечивающими корреляцию между связанными событиями.
3. **Сохранение последовательности событий** в пределах одного логического потока (conversation).
4. **Гарантию доставки** на уровне не ниже at-least-once для событий, инициирующих действия агентов.

Реализация **SHOULD** обеспечивать:

1. Семантику exactly-once для событий, влияющих на состояние физических активов или формирующих юридически значимые записи.
2. Возможность переигрывания (replay) событий в пределах настраиваемого окна времени для целей восстановления состояния и аудита.

## Допустимые транспорты

Спецификация не предписывает конкретный транспорт. Транспорт **MUST** обеспечивать гарантии, указанные в обязательствах.

Примеры допустимых транспортов: Apache Kafka, NATS, RabbitMQ при соответствующей конфигурации. Шины сообщений с гарантиями ниже at-least-once допускаются только для некритичных подсистем (телеметрия, метрики).

## Запрещённые паттерны

Реализация **MUST NOT** строить взаимодействие автономных агентов исключительно на синхронных HTTP-вызовах в режиме «request–response» без событийной декомпозиции. Использование синхронных интерфейсов допустимо только как инструмент адаптации к легаси-системам через адаптеры.
