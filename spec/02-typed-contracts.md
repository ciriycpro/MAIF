# 02. Типизированные контракты

## Требование

Все события, обмениваемые между агентами в межагентном контуре, **MUST** описываться типизированными контрактами. Контракт **MUST** быть машиночитаемым и доступным для верификации до и во время выполнения.

## Минимальные обязательства реализации

Каждый контракт **MUST** содержать:

1. **Идентификатор типа** сообщения, уникальный в пределах семантической области.
2. **Источник** сообщения — идентификатор агента-отправителя и его роль.
3. **Уровень критичности** сообщения — категория, определяющая регламент обработки.
4. **Права доступа** — перечень допустимых получателей либо политика доступа.
5. **Допустимые действия** — перечень операций, инициируемых сообщением.
6. **Формат полезной нагрузки** — схема данных в машиночитаемой нотации (JSON Schema, Protobuf, Avro или эквивалент).
7. **Версию контракта** — для обеспечения обратной совместимости.

## Эволюция контрактов

Реализация **MUST** обеспечивать механизм версионирования контрактов с правилами:

- Мажорная версия — несовместимое изменение.
- Минорная версия — обратно совместимое расширение (добавление опциональных полей).
- Реализация **MUST** отвергать сообщения, не соответствующие зарегистрированному контракту.

## Реестр контрактов

Реализация **SHOULD** предоставлять реестр зарегистрированных контрактов с возможностью их обнаружения и валидации во время выполнения.

## Связь с другими требованиями

- Сквозной аудит (раздел 05) опирается на типизированные контракты для верификации содержания событий в аудиторском следе.
- Human-in-the-loop (раздел 04) использует контракты для описания запросов подтверждения.
