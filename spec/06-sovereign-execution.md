# 06. Суверенное исполнение

## Требование

Реализация стандарта **MUST** обеспечивать исполнение критических компонентов в управляемом периметре. Принятие решений автономными агентами **MUST** происходить без runtime-зависимости от внешних сервисов и инфраструктур, находящихся вне контроля правообладателя данных и оператора системы.

## Определение управляемого периметра

Управляемый периметр — программно-аппаратная среда, в которой:

- Контроль над исполняемым кодом и его обновлением принадлежит оператору.
- Контроль над данными, обрабатываемыми системой, принадлежит правообладателю данных.
- Сетевое взаимодействие с внешними системами осуществляется только через явно разрешённые каналы.

## Минимальные обязательства реализации

1. **Локальное исполнение принятия решений.** Логика принятия решений автономным агентом **MUST** быть исполнима в управляемом периметре без обращений к внешним сервисам, недоступным в этом периметре.
2. **Локальное хранение состояния.** Состояние агентов и аудиторский след (раздел 05) **MUST** храниться в управляемом периметре.
3. **Воспроизводимость сборки.** Реализация **SHOULD** обеспечивать воспроизводимую сборку из исходных кодов в пределах управляемого периметра.
4. **Контроль зависимостей.** Реализация **MUST** документировать перечень внешних зависимостей и обеспечивать возможность их размещения в управляемом периметре.

## Допустимое использование внешних сервисов

Внешние сервисы (включая облачные LLM, внешние API данных, внешние сервисы аутентификации) **MAY** использоваться при выполнении условий:

- Использование не является runtime-зависимостью для принятия критических решений.
- Использование явно зафиксировано в политике агента и доступно для аудита.
- Отказ внешнего сервиса не приводит к нарушению базовой функциональности агента.

## Соответствие отраслевым требованиям

Реализация **MUST** обеспечивать возможность развёртывания в режиме, удовлетворяющем требованиям к программному обеспечению для объектов КИИ согласно действующему законодательству Российской Федерации.
